## 実行手順

### 1. 事前確認
- 何を自動化するか（現状の手動フローを具体的に把握）
- 手動だと何分かかるか（自動化の価値を測る）
- 既存の関連スクリプトがないか確認
- 実行環境: Windows? WSL2? サーバー?

### 2. 設計
- 最小限の実装から始める（MVP → 改善）
- 入力→処理→出力を明確にする
- エラーケースを先に洗い出す
- 設定値はハードコードせず、環境変数 or 設定ファイルに外出し

### 3. 実装規約
- 変数名・関数名は意図が伝��る命名（省略しない）
- コメントはWhy（なぜ）のみ。What（何をしているか）はコードで表現
- 1関数1責務
- 不要な���象化をしない（3回同じコードが出るまで共通化しない）

### 4. テスト
- 外部APIとの結合部分は実際に叩いて確認（モックに頼らない）
- テストをスキップしたら不合格（SKIP = FAIL）
- 2回実行しても壊れないことを確認（冪等性）

### 5. セキュリティ
- 秘密情報は.envに格納（ハードコード禁止）
- OWASP Top 10を意識
- ユーザー入力は必ずバリ���ーション

## 禁止パターン
- sudo / rm -rf / git push --force（明示的指示なしに実行しない）
- 秘密情報のハードコード
- エラーを握りつぶすこと（catch して何もしない）
- テストなしで「動くから大丈夫」と判断すること
- 絵文字の使用

## 品質基準

- [ ] エラー時にわかりやすいログが出るか
- [ ] 2回実行しても壊れないか（冪等性）
- [ ] 秘密情報がハードコードされていないか
- [ ] テストが実行され、パスしているか
- [ ] 使い方がREADME or コメントでわかるか
- [ ] 絵文字が使われていないか
